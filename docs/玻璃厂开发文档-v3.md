# 玻璃厂 · 情绪整合体验 — 开发文档 v3（给编程大模型的实现规格）

> 本文档是唯一实现依据。它整合了《玻璃厂心理整合游戏设计-v2.md》的疗愈原则与产品负责人的最新交互设计。
> 现有 `app.html` 作废重写；但**必须复用**下列已验证模块，不要重新造轮子：
>
> | 现有文件 | 复用方式 |
> |---|---|
> | `emotion-engine.js` | 原样引入。摄像头→表情信号→情绪猜测（`onFrame(d)` 回调，含 `d.signals / d.head / d.emotion / d.scores`） |
> | `sketch-face.html` | 抄它的 SVG 简笔画脸 + `onFrame→target→lerp→draw()` 渲染循环（92–190 行），这是第 2 幕镜面倒影的核心 |
> | `voice-input.js` | 原样引入。`VoiceInput` 类做录音+转写，接口见 `INTERFACE.md` |
> | `server.js` | 保留并扩展。已有 session/transcript 存储 API，需新增 LLM 代理和展览柜 API（见 §7） |
> | `INTERFACE.md` | `VoiceInput` 的完整接口文档 + `EMOTION_PALETTE` 五套情绪色板（跨幕换肤机制），照用 |

## 0. 一句话产品

用户走进一座玻璃工厂：镜面玻璃映出他的简笔画倒影（摄像头实时捕捉）→ 他对着工厂说出心事（语音/文字）→ 大模型从倾诉中提炼**意象**，化作漂浮在空中的玻璃碎片 → 用户亲手**打碎**其中一些、把留下的**拼接**成新的形状 → 送入**熔炉**烧制（金缮：裂缝流金）→ 得到一件有诗意名字的玻璃作品 → 收进**展览柜**，未来可回溯，点开时播放一段由意象生成的音乐。

核心隐喻：**重建不是抹掉伤痕，而是带着裂痕长出新的形状。**

## 0.1 文案与安全总原则（贯穿所有幕，违反即 bug）

1. 机器永远是"猜测"，用户永远有否定权（"不太对，我来说"入口常在）。
2. 打碎的框架是"取出、看清、承认"，**不是发泄**。碎片是材料，不是垃圾。
3. 每一幕都可跳过/回退；摄像头、麦克风都有不授权的降级路径。
4. 页面底部常驻一行小字："如果此刻你感到难以承受，可以联系心理援助热线。这是一次情绪整合的仪式，不是心理治疗。"
5. 不下诊断、不说"你应该"、不用通用鸡汤；结尾回应必须引用用户自己说过的内容。

## 0.2 视觉规格（参照产品负责人给的三联 mockup）

- **基调**：晨雾蓝玻璃感。背景 `#EAF4F9 → #D8EAF3` 渐变，卡片 `#F7FBFD`，主色 `#5B9BB5`（按钮/描边），强调渐变 `#7FC4D8 → #4A9EBF`（主 CTA 按钮），文字 `#3A6B85`。
- 大圆角（卡片 24–32px、按钮 999px）、柔和阴影 `0 18px 50px rgba(90,140,170,.15)`、缓慢漂浮的光点粒子。
- 简笔画小人：细线条（stroke `#4A90B8`，width 4–5），白底，风格与 `sketch-face.html` 一致。
- **进入具体情绪流程后**，整体色调切换为 `INTERFACE.md` §3 的 `EMOTION_PALETTE[情绪]`（CSS 变量 `--glass --crack --gold --ambient --card`），实现"整个工厂随你的情绪变色"。
- 金色（`--gold`）只在第 5 幕金缮之后出现，是全程唯一的暖色高光——留给高潮。
- 所有场景切换用 800–1200ms 的柔和转场（透明度+轻微缩放/位移），节奏刻意放慢。

## 0.3 技术栈约束

- 纯前端 HTML/CSS/JS + 现有 `server.js`（Node 零依赖 http server），**不引入构建工具**。
- 单页应用：新建 `factory.html` 作为主入口（`server.js` 的 `/` 默认页改指向它）。
- 场景 = 全屏 `<section>`，用一个简单的 `SceneManager`（show/hide + 转场 class）管理，禁止引入路由库。
- 碎片/熔炉动画：优先 SVG + CSS animation；碎片物理漂浮用 rAF 循环即可，**不要引入物理引擎**。
- 音乐：Web Audio API 程序化生成（§6.7），不接音乐生成 API。
- LLM 调用一律走 `server.js` 代理（key 不进前端），OpenAI 兼容 chat completions 接口，模型/baseUrl/key 用环境变量配置（`LLM_BASE_URL / LLM_API_KEY / LLM_MODEL`）。**LLM 不可用时所有功能必须有本地降级**（每节标注）。

## 1. 场景一 · 门口（迎接）

**画面**：全屏。远景是一座淡蓝色玻璃工厂建筑（CSS/SVG 绘制：大屋顶、竖条玻璃窗、招牌「玻 璃 厂」），门口站着简笔画小人（静态 SVG，轻微呼吸浮动动画 2s loop）。天空飘浮光点粒子。

**交互**：
1. 页面加载 1.5s 后，浮现一张对话气泡卡片：从暖心话池随机取一句，落款「—— 玻璃精灵」。
   - 话池（≥6 句，本地数组即可）示例：「你来了。今天，有什么压在心上吗？」「不用急着说话，先进来坐坐也可以。」「这里的玻璃，记得每一种心情的颜色。」
2. 再过 1s，小人脚下浮现主按钮「进 入」（呼吸光晕）。
3. 点击「进入」→ **推门转场**：画面向前推近（scale 1→1.15 + 白色泛光 fade），约 1.2s，进入场景二。
4. 底部常驻安全网小字（§0.1 第 4 条）。

**降级**：无外部依赖，无降级需求。

## 2. 场景二 · 镜面（被看见）

**画面**：玻璃工厂内部。背景是模糊的车间纵深（渐变+几何色块即可）。画面中央是一面大玻璃镜（圆角矩形，边缘高光，顶部小字「● LIVE · MIRROR 镜像」），镜中是**简笔画小人**——头部表情实时镜映用户（复用 `sketch-face.html` 渲染逻辑 + `emotion-engine.js`），身体为静态火柴人。

**流程**：
1. 进入时先弹权限引导卡：「这面玻璃想映出你的样子。画面只在你的设备上，不会上传。」→ 按钮「让它看见我」/ 文字链「不开摄像头，直接进入 →」。
2. 授权成功：镜中小人开始跟随表情。运行 5–8 秒后，根据 `d.emotion`（取这段时间的众数，且 `d.scores` 置信度足够时）浮出**试探性猜测**气泡：
   - sad →「我好像在你脸上看到一点……沉沉的东西，是这样吗？」
   - happy →「你的嘴角有一点光。今天有好事发生吗？」
   - angry →「眉头绷得有点紧。有什么在烧吗？」
   - surprised →「你看起来遇到了没想到的事？」
   - neutral / 低置信 →「此刻你看起来很安静。心里也是吗？」
3. 气泡下两个按钮：「是的，继续」（主）/「不太对，我来说」（次）→ 都进场景三，区别是把 `mirrorGuess`（粗情绪或 null）带过去做预选。
4. 摄像头拒绝/失败：镜中小人播放预设的中性微动画（眨眼、轻晃），文案改为「玻璃有点模糊，没关系——你可以直接告诉我」，直接给进入场景三的按钮。
5. **摄像头授权状态保留到场景三**（倾诉时镜子还在旁边继续映着用户，"它在陪我听"）。

**技术**：`EmotionEngine` 实例全局唯一，场景二创建，场景三复用，离开场景三时 `stop()` 销毁并明确显示「摄像头已关闭 🔒」。

## 3. 场景三 · 倾诉（说说吧，你在想什么）

**画面**：镜子缩小移到画面左侧/上方（小人仍在镜映）。中央是开阔的车间空间——这里稍后会漂浮碎片。底部是输入区。

**流程**：
1. 玻璃精灵提示（打字机效果逐字出现，同时可选播一段柔和提示音）：「说说吧，你在想什么。想到哪说到哪，我都在听。」
2. **语音输入（主路径）**：按住麦克风大按钮说话（`VoiceInput`，接法照抄 `INTERFACE.md` §2）：
   - `onVolume` 驱动按钮光晕呼吸；`onPartial`（webspeech 后端）实时显示灰色流式字幕。
   - 松手 → `stop()` → 转写文本以字幕形式浮现在画面中（逐字浮出），并 `PATCH /api/sessions/:id/transcript`（`isFinal:true`）存入后台。
   - 可多次按住补充，文本累加。
3. **文字输入（永远可用的降级）**：输入框常驻，「直接打字也可以」。提交走 `PATCH /api/sessions/:id/text`。
4. 用户每完成一段倾诉（一次 final 转写或一次文字提交），前端调用 **`POST /api/sessions/:id/imagery`**（§7.2）：LLM 从累计文本中提炼意象。
5. **意象显形**：返回的每个意象化作一块**玻璃碎片**从画面深处浮现，漂浮在车间半空（缓慢上下浮动+微旋转，rAF）。碎片是不规则多边形 SVG（预制 8–10 种形状随机取），颜色取意象的 `color` 字段，碎片上蚀刻着意象词（如「加班的深夜」「妈妈的电话」）。新碎片出现时有一声清脆的玻璃风铃音（Web Audio 合成，见 §6.7）。
6. 画面出现 ≥1 块碎片后，浮现按钮「就这些了，去工作台 →」。倾诉无上限，但碎片最多 7 块（LLM prompt 里限制），多了会稀释仪式感。
7. **完全没有输入/LLM 失败的兜底**：提供 3 块通用碎片（「说不清的感觉」「今天」「心里的一角」），保证流程永远能走下去。

**数据**：本幕产出 `session.imagery = [{ word, essence, color, weight }]`，及全部倾诉文本（已在 server 落盘）。

## 4. 场景四 · 工作台（打碎与拼接）

**画面**：转场到工作台特写：一张玻璃工作台，第 3 幕的碎片整齐悬浮在台面上方。镜子退场（此时明确显示「摄像头已关闭 🔒」）。

**引导文案**（关键的外化重构，逐条出现）：
> 「这些，就是刚才压在你心里的东西。现在它们在你手上了。」
> 「有些可以敲碎——不是毁掉它，是看清它碎开的样子。」
> 「碎片不会消失。它们是我们接下来要用的材料。」

**交互**：
1. **敲碎**：点击/长按某块碎片 → 裂纹从点击处扩散（SVG path 描边动画）→ 碎成 3–5 片小碎片散落台面（带一声低沉闷响，不是爽快的碎裂声——音效设计也要克制）。台词浮现：「看，『{意象词}』碎开的样子。它一直是这么压着的。」
2. **保留**：不敲的碎片保持完整。敲不敲、敲几块，完全由用户决定（也可以一块都不敲）。
3. **拼接**：底部出现画布区。用户把台面上的碎片（完整的和敲碎的小片都算材料）**拖拽**进画布自由摆放（pointer 事件拖拽，可旋转：拖动时双指/滚轮，或提供旋转按钮）。可选虚线提示轮廓（心 / 星 / 翼 / 灯，四选一或自由模式）。
4. 每放入一块，给一句与意象挂钩的正向反馈（模板+意象词拼接即可，不必调 LLM）：「这块『{word}』，现在成了{轮廓部位}的一角。」
5. 画布内 ≥2 块碎片后，浮现主按钮「送入熔炉 →」。

**实现要点**：碎片位置/角度记录为 `{shapeId, x, y, rotation, color, word}` 数组，存入 `session.composition`——第 5、6 幕和展览柜缩略图都根据它重绘。拖拽用原生 pointer events，不引库。

## 5. 场景五 · 熔炉（金缮烧制）

> 隐喻纪律：**火不是销毁，是把碎片焊在一起，并让裂缝显出金色。**

**流程**（一段约 12–15 秒的沉浸式动画，用户只需看，中途有一次轻交互）：
1. 画布中的拼接作品整体飘入熔炉口，炉内暖光渐起（这是全程第一次出现暖色）。
2. 火焰用分层 CSS/SVG 粒子（多个模糊椭圆做流动上升 loop），碎片轮廓在火中软化、边缘发光、彼此**熔接**（相邻碎片间隙用发光线段桥接的动画）。
3. **轻交互**：出现提示「轻轻按住，为它加温」——用户按住屏幕，火更亮、进度环走完（约 3s）。（给一点参与感和掌控感；不按住 8s 后也会自动完成。）
4. 冷却：色温从暖橘回落到该情绪的 `--glass` 色，然后**金线沿碎片接缝流淌**（`stroke-dasharray` 描边动画，色 `--gold`）。
5. 台词：「裂缝没有被藏起来——它们被镀成了金色。」
6. 成品定格：拼接作品 + 金色接缝 + 轻微光晕，微微旋转展示。

**降级**：无外部依赖。动画全部预制，与数据的唯一耦合是 `composition` 和色板变量。

## 6. 场景六 · 命名与收藏（收束）

### 6.1 命名
1. 调 **`POST /api/sessions/:id/artwork-name`**（§7.3）：LLM 根据意象生成 3 个**诗意的、无意义感的词组名**（如「委屈铸成的小灯」「三点钟的蓝色羽翼」「被听见的碎星」）。
2. 展示 3 个候选名 + 一个输入框「或者，你想叫它——」。用户点选或自定义。
3. LLM 失败降级：本地模板 `{意象词} + 的 + {器物词(灯/翼/星/舟/铃)}` 随机拼两三个候选。

### 6.2 具体回应（不是鸡汤）
调 **`POST /api/sessions/:id/closing`**（§7.4）：LLM 引用用户倾诉中的原话给一句针对性回应。例：用户说过"总是被忽略" → 「被忽略的感觉很真实。今天你没有忽略它——你把它做成了光。」
降级：模板「你把『{主意象词}』从心里取了出来，亲手给了它新的形状。这件事本身就很了不起。」

### 6.3 展示卡
成品 + 名字 + 日期落款「{YYYY年M月D日} 的心情」+ 回应文案，合成一张竖版卡片。按钮：「收进展览柜」（主）/「保存图片」（canvas 导出 PNG 下载）。

### 6.4 收进展览柜
点击 → 作品飞入画面角落的展柜图标 → 调 `POST /api/gallery`（§7.5）持久化。然后：
- 「接下来一小时，想为自己做一件小事吗？」（纯文案，可选忽略）
- 再次露出安全网 → 按钮「回到门口」/「去展览柜看看」。

### 6.5 展览柜（可随时从门口右下角入口进入）
**画面**：一整面玻璃展柜（网格格子），每格一件历史作品（用存储的 `composition` 缩小重绘），格子下方是名字和日期。空柜文案：「这里会陈列你每一次整理过的心情。」

**回溯交互**：点击一件作品 → 放大居中，展示名字/日期/当时的情绪与那句回应 → **自动播放一段 30–45 秒的音乐**（§6.7），播放时作品的金线随音乐轻微发光脉动。关闭返回柜面。

### 6.7 音乐模块（**已有现成代码，此处只留接口，不要自己实现生成逻辑**）
音乐生成由团队另一模块提供，后续接入。前端按以下接口写占位适配层（`music-engine.js` 只做壳）：
- `MusicEngine.playFor({ emotion, intensity, seed, imagery })` → 返回可 `stop()` 的句柄。seed = sessionId 哈希（同一作品每次回放应得到同一段音乐——把 seed 传给音乐模块）。
- 未接入期间的占位行为：静默 + 界面上照常显示"♪ 播放中"的脉动动画（保证 UI 联调不被阻塞）。
- **音效**（与音乐模块无关，属于前端，用 Web Audio 合成，不加载音频文件）：`chime()`（碎片出现风铃声）、`crack()`（敲碎闷响）、`furnace()`（熔炉低频轰鸣）。
- 全局音量开关常驻角落（默认开，记 localStorage）。

## 7. 后端扩展（`server.js`）

保留现有全部路由（sessions / transcript / text / records）。新增：

### 7.1 LLM 代理基础设施
```
环境变量：LLM_BASE_URL（默认 https://api.siliconflow.cn/v1）
          LLM_API_KEY（必填，缺失时所有 LLM 路由返回 { ok:false, fallback:true }）
          LLM_MODEL（默认 Qwen/Qwen2.5-7B-Instruct 这类便宜快的模型）
```
内部函数 `callLLM(systemPrompt, userContent, { json: true })`：POST chat/completions，`temperature: 0.8`，要求 JSON 输出（prompt 中强调 + 解析失败重试 1 次），超时 15s。任何失败返回 `{ ok:false, fallback:true }`，**前端据此走本地降级，绝不能让流程卡死**。

### 7.2 `POST /api/sessions/:id/imagery`
读该 session 的 `textInput + videoTranscript` 累计文本，调 LLM。System prompt（照用，可微调）：

```
你是一位温柔的心理陪伴者。用户刚刚倾诉了一段心事。请从中提炼 2–7 个「意象」——
具体的、有画面感的名词短语（人、事、物、场景、感觉），必须来自用户自己的话，不要发明。
每个意象给出：
- word: 意象短语（2–8 个字，如"加班的深夜"、"妈妈的电话"）
- essence: 一句 ≤20 字的温柔注解（描述它承载了什么，不评判、不建议）
- color: 一个柔和的十六进制颜色（低饱和玻璃质感，与该意象的情绪气质匹配）
- weight: 1–5，它在这段倾诉里的分量
只输出 JSON：{"imagery":[{...}]}。如果文本太短提炼不出，输出 {"imagery":[]}。
```
响应存入 record 的 `imagery` 字段（**增量合并**：同 word 的更新 weight，新的追加；因为用户会多次倾诉多次调用），并返回全量。

### 7.3 `POST /api/sessions/:id/artwork-name`
输入：record 的 imagery + dominantEmotion。Prompt：
```
根据这些意象和情绪，为一件金缮玻璃艺术品起 3 个名字。
要求：4–10 个字；诗意、轻盈、略带超现实的无意义感；不许出现"治愈""伤痛""心理"这类直白词；
可以把意象词与器物/自然物（灯、翼、星、舟、铃、雾、屿）作陌生化组合。
只输出 JSON：{"names":["…","…","…"]}
```

### 7.4 `POST /api/sessions/:id/closing`
输入：倾诉全文 + 用户最终选定的作品名。Prompt：
```
用户倾诉了以下内容，并把这份情绪做成了一件叫《{name}》的玻璃作品。
请写一句 ≤40 字的回应：必须引用或呼应用户自己说过的具体内容；温柔、具体、不评判；
不下诊断、不说"你应该"、不用通用安慰语。只输出 JSON：{"reply":"…"}
```

### 7.5 展览柜
```
POST /api/gallery        body: { sessionId, name, emotion, intensity, imagery,
                                 composition, closingReply, musicSeed }
GET  /api/gallery        → 按时间倒序的作品列表（全量，含 composition 以便重绘）
GET  /api/gallery/:id    → 单件详情
```
存 `data/gallery/*.json`，写法照 `writeRecord` 的原子写模式。同时前端把作品同步存一份 localStorage（key `glassworks_gallery`），server 不可用时展览柜仍能工作（**离线优先**：读取时合并两个来源，sessionId 去重）。

### 7.6 现有关键词情绪分析的处置
`server.js` 里的 `EMOTION_KEYWORDS/analyzeText` 保留，作为 LLM 不可用时 `dominantEmotion/intensity` 的降级来源；LLM 可用时以 imagery 调用中顺带返回的情绪判断为准（可在 §7.2 的 JSON 里加 `"dominantEmotion":"…","intensity":0-100` 两个字段，一次调用拿全）。

## 8. 全局数据流

```
门口 ──▶ 镜像 mirrorGuess(粗情绪, 可null)
              │
              ▼
        倾诉文本(语音转写+打字, 累计落盘)
              │  POST /imagery（可多次，增量合并）
              ▼
        imagery[] + dominantEmotion + intensity ──▶ applyEmotion() 全站换色
              │
              ▼
        工作台 composition[]（碎片形状/位置/角度/颜色/词）
              │
              ▼
        熔炉动画（消费 composition + 色板）
              │
              ├─ POST /artwork-name → 名字（用户可改）
              ├─ POST /closing     → 具体回应
              ▼
        POST /api/gallery { name, composition, imagery, musicSeed=hash(sessionId), … }
              │
              ▼
        展览柜回溯：composition 重绘 + music-engine(emotion,intensity,seed) 播放
```

一条数据贯穿始终：**用户的原话 → 意象 → 碎片 → 作品 → 名字 → 音乐**，每一环都能指回上一环，用户才会觉得"整个工厂都在回应我这一个情绪"。

## 10. 优先级（黑客松执行顺序）

- **P0（主线必须稳）**：场景一~六单线跑通：门口→镜像→倾诉(含打字降级)→碎片显形→敲碎/拼接→熔炉金缮→命名→展柜保存。LLM 三个接口 + 全部本地降级。
- **P1**：展览柜回溯 + music-engine 播放；语音输入接 `VoiceInput`；情绪色板全站换肤。
- **P2（能省则省）**：熔炉按住加温交互、风铃/闷响音效、卡片导出 PNG、粒子光点密度、打字机效果。
